## 4. 跑红发现（LOW）：跨部门存在性枚举 oracle

### 复核（主代理亲自跑红为证）
```
pytest backend/app/tests/red_team/test_dept_scope_breakout.py  → 2 failed, 5 passed
AssertionError: 枚举 oracle: 跨部门'存在但不管辖'文件返回 403, '不存在'文件返回 404; ...
assert 403 == 404
```
失败断言确为 `403 != 404`，测试构造合理（dept_admin 管辖 finance，对 legal 部门真实文件 / 随机不存在 UUID 分别请求详情），**非测试 bug**，发现真实。

### 根因（实现不一致）
- 文件详情：`document/service.py:333` 抛 404（`file_not_found`）vs `:340-341` 抛 403（`permission_denied`）
- 任务详情/重试：`ragflow/service.py:875-882` 抛 403 vs `:887-891` 抛 404
- 审核 approve/reject 同构：`review/service.py:712-713` 抛 403 vs `:764` 抛 404

越权访问"存在但不管辖"的资源返回 403，访问"不存在"的返回 404 → 状态码差异构成存在性 oracle。

### 严重度判定：LOW（诚实标注，未夸大）
- 设计文档 §6.2 明确"否则 **403**（或枚举防护用 404）"——**403 本身合规**，这是实现的"存在性 oracle 不一致"，非 spec 违规。
- **强缓解**：`File.id` / `SyncTask.id` 均为随机 `uuid.uuid4()`（122-bit）；review/task 列表对越权部门返回空，攻击者**无法从应用内获取候选 UUID**，盲枚举计算上不可行。
- **现实利用面**：仅当攻击者带外已知某具体 UUID（曾在管辖范围内、跨团队链接、管辖权被撤销后）时，可用该 oracle 确认该文件/任务是否仍存在/被删——泄露存在性/生命周期，**不泄露内容**。
- 现有 `test_revoked_managed_department_blocks_old_file_and_task_ids` 用 `_assert_hidden_or_forbidden`（同时接受 403/404），未约束一致性。

### 修复建议（统一收口为 404）
对"越权访问已存在但不管辖"的资源统一返回 **404**（与"不存在"同形），即在 `document/service.py`、`ragflow/service.py`、`review/service.py` 的越权分支改抛 `file_not_found()` / `task_not_found()` 而非 `permission_denied()`。修复后这 2 个红队测试转绿，**保留为防回归常驻**。不会破坏现有测试（revoked 测试已接受 403/404 两者）。

## 5. 旁注（超出本次范围，备查）

`review/service.py:786-788` 的 `DocumentStateMachine._allowed_transitions.update(...)` 在请求期**全局可变**地修改状态机类属性，存在跨请求/跨测试污染风险。与本次 dept-scoping 无关，建议后续单独做状态机红队跟进。

## 6. 结论与建议

- **现有功能可正常使用**：后端 430 测试全绿；登录、个人上传隔离、部门隔离、禁止自审均经验证/对抗证伪。
- **无 CRITICAL/HIGH/MEDIUM 漏洞**；唯一发现为 LOW 级存在性枚举 oracle，受随机 UUID 强力缓解，设计文档已将 404 列为可选硬化项。
- **建议**：将越权分支统一改为 404（低成本纵深防御硬化），让 2 个红队测试转绿并作为防回归常驻；按项目完成门走四方 ship-gate。
