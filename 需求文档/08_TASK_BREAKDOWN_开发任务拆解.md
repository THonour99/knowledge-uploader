## 2. 当前五个工作流

### A. 基线恢复（P0）

- 恢复 PRD、架构、后端、前端、数据库/API、AI/RAGFlow、部署与任务文档。
- 05 文档成为唯一状态/API 权威源；README 去除“阶段 9 已完成”。
- 建立 IA、配置契约与可执行验收矩阵，修复所有交叉引用。

### B. 主链修复（P0）

- 部门注册/未分配门禁、邮箱验证与重置边界。
- `uploaded=草稿`、手工/自动提交、AI 完成续接、critical 阻断、驳回重提。
- 鉴权原件 preview/download；明确 `sync|approve_only` 与 Dataset 必填。
- 权限、审计、并发和迁移负例齐全。

### C. 产品工作台（P1）

- `/dashboard` 角色自适应；员工待办、管理员队列与系统健康可下钻。
- 审核领取/SLA、通知闭环、服务端分页搜索、URL 状态、移动抽屉布局。
- 前端组件/状态/tokens 统一，关键交互与可访问性测试。

### D. 配置与上线门禁（P1）

- 清零死配置，完成逐 key 行为测试。
- 真实 PostgreSQL/RabbitMQ/Redis/MinIO + workers E2E 和故障注入。
- DLQ、指标告警、runbook、PostgreSQL+MinIO 配对恢复。
- buildx 预检 + DGX Spark ARM64 实机证据。

### E. 增强治理（P2）

- 真正 LLM 分析和可审计 token/成本。
- 文档版本/替代、到期负责人、保存视图。
- 容量和成本指标、隐私与预算门禁。

## 3. 合并顺序

1. 先合并 A 与 B 的兼容后端契约和迁移。
2. C 基于稳定 API 接入；D 可与 C 并行但发布门禁最后汇总。
3. P0/P1 全通过后再默认开启 E；P2 迁移必须向后兼容。
4. 每个原子变更独立 commit，随后由非作者审查权限、状态、迁移、配置、测试和文档一致性。

## 4. 任务完成模板

每项任务必须记录：负责人、关联需求/矩阵 ID、修改路径、迁移、测试命令与结果、风险/回滚、reviewer、commit SHA。只写“已实现”而没有验收 ID 和证据，不算完成。
