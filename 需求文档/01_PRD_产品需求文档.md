# 01. Knowledge Uploader 产品需求文档

> 版本：2.0
> 状态：权威基线（2026-07-16）
> 产品范围：员工贡献文档、部门审核、RAGFlow 同步与治理
> 关联契约：[状态机与 API](./05_DATABASE_API_SPEC_数据库与API规范.md) · [信息架构](../docs/product/IA_ROLE_WORKBENCH.md) · [配置契约](../docs/product/CONFIG_CONTRACT.md) · [验收矩阵](../docs/product/ACCEPTANCE_MATRIX.md)

## 1. 产品目标

Knowledge Uploader 解决“谁提交了什么、谁批准进入哪个知识库、当前处理到哪里、出现问题如何追责”四个问题。员工上传的原件必须先归属到确定部门，再经过可选 AI 分析与人工审核，只有管理员做出明确同步决定后才能进入 RAGFlow Dataset。

本轮交付不是“页面存在即完成”。完成必须同时满足：业务主链可执行、权限隔离成立、失败可恢复、配置确有运行时消费者、真实基础设施 E2E 通过、ARM64 实机与备份恢复有证据。

## 2. 角色与数据边界

| 角色 | 数据范围 | 核心能力 | 禁止事项 |
|---|---|---|---|
| 员工 `employee` | 自己上传的文档与通知 | 上传、保存草稿、提交审核、查看进度、预览/下载自己的原件、处理驳回 | 查看他人文档；选择 RAGFlow Dataset；审批 |
| 部门管理员 `dept_admin` | 被授权部门及下级部门（若配置）的文档 | 领取审核、查看原件/分析、调整元数据、批准/驳回、明确是否同步 | 越部门访问；管理系统密钥；静默改变同步目标 |
| 系统管理员 `system_admin` | 全局 | 用户/部门/配置/Dataset/审计/运维治理；可处理跨部门异常 | 绕过审计；向前端或日志暴露密钥 |

所有管理员读取他人原件、领取/释放、审批、配置、重试、用户和部门变更都必须写审计日志。前端权限只改善体验，后端必须逐请求鉴权与数据域过滤。

## 3. 账号、验证与部门归属

### 3.1 注册

- 注册请求必须包含一个当前启用的 `department_id`。后端校验后直接写入用户部门；名称文本不能代替外键。
- 若兼容旧客户端而未提供部门，账号可创建但标记为“未分配部门”；未分配用户不能上传或提交文档，工作台只显示补全提示。
- 自选部门不会授予管理员权限；角色只能由系统管理员授予。
- `security.require_email_verification=true` 时，注册账号处于待验证门禁。验证 token 原文只发送一次，数据库、outbox 与日志只能保存 SHA256 hash 或不可重放引用。
- 验证关闭时账号可直接激活，但仍受部门门禁约束。

### 3.2 登录与恢复

- 待验证账号不得获得 JWT；返回稳定错误码 `EMAIL_NOT_VERIFIED`，前端提供重发入口。
- 密码重置只改变凭据，绝不能顺带把未验证账号激活。
- 禁用、锁定、未验证和未分配部门是不同状态/门禁，界面必须给出不同恢复动作。
- 邮件链接必须落到真实前端路由：`/verify-email?token=...` 与 `/reset-password/:token`。

## 4. 文档主链

### 4.1 上传与草稿

1. 员工通过扩展名、实际 MIME、大小、配额、速率、文件名与 SHA256 去重校验。
2. 原件写入 MinIO，元数据与 `FileUploaded` outbox 在数据库事务内提交。
3. 持久化状态 `uploaded` 的产品标签统一为“草稿（已上传）”；不再引入另一个含义重叠的 `draft` 状态。
4. 员工可在草稿态补充标题、说明、可见性、到期日和负责人，然后手工“提交审核”。
5. `submit_after_upload=true` 表示用户显式选择自动提交，不是全局静默提交。

### 4.2 AI 开关与自动提交

- AI 关闭：文件不得进入任何 AI 状态。手工提交或 `submit_after_upload=true` 直接 `uploaded -> pending_review`。
- AI 开启：`uploaded -> extracting_text -> analysis_queued -> analyzing`，成功后进入 `analyzed`，发现需人工确认的敏感内容进入 `sensitive_review_required`，失败进入 `analysis_failed`。
- 自动提交开启时，普通分析成功后自动 `analyzed -> pending_review`；`critical` 必须停在敏感复核，不能自动提交或同步。
- AI 失败是否允许提交由单一策略决定；允许时界面和审核记录必须明确标注“AI 分析失败后提交”，不允许时提供重试。

### 4.3 审核

- 待审核文件进入部门队列并生成 `submitted_at`、`review_due_at`。默认 SLA 以配置为准。
- 部门管理员必须先领取；同一文件只允许一个有效领取人。并发领取返回 409，超时可自动释放，系统管理员可强制转派并写原因。
- 管理员必须能够在详情中查看原件、上传信息、版本关系、AI 分析、敏感命中和审计时间线。
- 驳回必须填写原因，员工修正后可从 `rejected` 重新提交；历史审核记录不可覆盖。

### 4.4 明确的 RAGFlow 决策

批准请求必须二选一，禁止以“可空 Dataset”暗示行为：

| `sync_decision` | Dataset | 结果 |
|---|---|---|
| `sync` | 必须提供或已有一个启用且有权限的 `dataset_mapping_id` | `approved -> queued`，创建唯一同步任务 |
| `approve_only` | 不需要 | 停留 `approved`，不创建同步任务；以后只能由有权管理员显式发起同步 |

`critical` 默认永远不能选择 `sync`。`high` 若允许同步，必须显示风险确认并记录理由。每个文件同一时间最多一个活跃同步任务，锁为 `lock:sync:{file_id}`。

## 5. 原件访问

- 原件通过受鉴权的后端端点流式返回，不暴露永久 MinIO URL 或对象凭据。
- 员工只能访问自己的原件；部门管理员只能访问管辖部门；系统管理员全局可访问。
- `inline` 仅用于浏览器安全预览；不安全类型强制附件下载。文件名按 RFC 5987 与 Windows 保留名规则清洗。
- 管理员读取他人原件写审计；响应支持合理的 `Range`、`ETag` 和 `Content-Length`，大文件不得整体读入内存。

## 6. 工作台与通知闭环

- `/dashboard` 是登录后的角色工作台：员工看到草稿、待审、驳回、同步结果和快捷上传；部门管理员看到未领取、我领取、临近/已超 SLA 和同步失败；系统管理员看到全局健康、积压、失败、容量与成本。
- 列表查询必须服务端分页、搜索和筛选；刷新、返回详情后保留查询条件，并允许 P2 保存视图。
- 状态变化生成站内通知；通知支持未读数、单条/全部已读和受控深链。点击通知必须落到用户有权访问的详情，否则显示已失效而不是泄露存在性。
- ≤768px 使用抽屉导航、单列筛选和卡片/横向滚动表格；主要动作不依赖 hover。

## 7. 安全与敏感文档

- 密码 Argon2id，JWT HS256 secret 至少 32 字节，API Key Fernet 字段加密。
- `critical` 默认阻止 RAGFlow；配置只能收紧或在受审计的专门流程中放宽 `high`，不得放宽 `critical`。
- 文件删除为软删；若涉及远端清理，任务失败进入可恢复状态且保留证据。
- 任何密钥不返回前端、不出日志、不进入通知正文；审计元数据也必须脱敏。

## 8. 治理增强（P2）

- 真正的 LLM 分析必须记录 provider/model/prompt 版本、输入摘要、耗时、token/估算成本和失败类别；规则分析不能冒充 LLM 结果。
- 新文件可声明 `replaces_file_id`，形成不可环版本链；替代生效前旧版本仍可用，生效后保留审计并按策略处理远端文档。
- 文档可设置 `owner_id` 与 `expires_at`；到期前通知负责人和部门管理员，逾期状态不等于物理删除。
- 容量与成本指标按部门、文件类型、处理阶段和时间聚合，不能泄露原文或密钥。

## 9. 优先级与完成定义

| 优先级 | 范围 | 完成定义 |
|---|---|---|
| P0 | 基线、账号/部门、主状态机、原件、明确同步决定 | 单元/API/E2E 通过，权限与审计负例齐全 |
| P1 | 角色工作台、领取/SLA、通知、分页搜索、移动端、上线门禁 | 真实 PostgreSQL/RabbitMQ/Redis/MinIO 链路及故障恢复有证据 |
| P2 | 真 LLM、版本、到期负责人、保存视图、容量成本 | 有可回滚迁移、指标口径、预算/隐私保护和验收用例 |
